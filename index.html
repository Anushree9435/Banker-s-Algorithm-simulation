01

Introduction to Deadlocks

Foundation Concept

A deadlock is a situation in a multiprogramming environment where two or more processes are blocked forever, each waiting for a resource held by another process in the set. No process can proceed because each is waiting for an event that can only be caused by another blocked process.

💡
Real-world Analogy: Imagine two cars meeting on a narrow bridge from opposite sides. Neither can move forward or backward — they're deadlocked. In computing, processes holding resources while requesting others create the same situation.

Deadlocks are a critical concern in operating systems because they can cause system-wide resource starvation. The system must either prevent, avoid, detect, or recover from deadlocks.

02

Four Necessary Conditions for Deadlock

Coffman Conditions (1971)

A deadlock can occur if and only if all four of the following conditions hold simultaneously:

1

Mutual Exclusion

At least one resource must be held in a non-shareable mode. Only one process can use the resource at a time. If another process requests it, it must wait.

2

Hold and Wait

A process must be holding at least one resource while waiting to acquire additional resources that are currently held by other processes.

3

No Preemption

Resources cannot be forcibly taken away from a process. A resource can only be released voluntarily by the process holding it after completing its task.

4

Circular Wait

There must exist a circular chain of two or more processes, each waiting for a resource held by the next process in the chain.

⚠️
Important: All four conditions must be present simultaneously for a deadlock. If we can prevent even one condition, deadlock becomes impossible.
03

Deadlock Handling Strategies

Prevention, Avoidance, Detection & Recovery

Strategy
Approach
Trade-off
Prevention
Ensure at least one Coffman condition cannot hold
Low resource utilization, reduced throughput
Avoidance
System dynamically examines resource-allocation state before granting
Requires advance information about maximum resource needs
Detection
Allow deadlocks, detect them, then recover
Recovery may involve process termination or rollback
Ignorance
Pretend deadlocks never occur (Ostrich Algorithm)
Used in many real systems (Linux, Windows) for simplicity
🎯
Banker's Algorithm falls under the Avoidance strategy. It ensures the system never enters an unsafe state by checking safety before granting any resource request.
04

Banker's Algorithm

Developed by Edsger Dijkstra (1965)

The Banker's Algorithm is named after a banking analogy: a banker will not allocate capital in such a way that it could no longer satisfy the needs of all its customers. Similarly, the OS acts as a banker for system resources.

Data Structures

Let n = number of processes, m = number of resource types:

Available
Vector [m]

Number of available instances of each resource type. Available[j] = k means k instances of resource Rj are available.

Maximum
Matrix [n×m]

Maximum demand of each process. Max[i][j] = k means process Pi may request at most k instances of resource Rj.

Allocation
Matrix [n×m]

Resources currently allocated. Allocation[i][j] = k means process Pi currently holds k instances of Rj.

Need
Matrix [n×m]

Remaining resource need. Need[i][j] = Max[i][j] − Allocation[i][j]. Number of resources Pi may still request.

05

Safety Algorithm

Core of the Banker's Algorithm

The safety algorithm determines whether a system is in a safe state. A safe state guarantees that all processes can complete in some order without deadlock.

1
Initialize: Work = Available (copy of available vector)
Finish[i] = false for all i = 0, 1, ..., n-1
2
Find: an index i such that:
Finish[i] == false AND Need[i] ≤ Work
If no such i exists, go to step 4.
3
Execute:
Work = Work + Allocation[i]
Finish[i] = true
Go to step 2.
4
Check: If Finish[i] == true for all i, then the system is in a safe state. Otherwise, it is unsafe.
06

Resource-Request Algorithm

Handling New Requests

When process Pi makes a request Request[i]:

1
If Request[i] ≤ Need[i], go to step 2. Otherwise, raise an error — the process has exceeded its maximum claim.
2
If Request[i] ≤ Available, go to step 3. Otherwise, Pi must wait — resources are not available.
3
Pretend to allocate requested resources:
Available = Available − Request[i]
Allocation[i] = Allocation[i] + Request[i]
Need[i] = Need[i] − Request[i]

Run the Safety Algorithm:
• If safe → grant the request
• If unsafe → deny the request, rollback, and Pi must wait
